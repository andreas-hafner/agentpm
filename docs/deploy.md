# Deploy

`agentpm deploy` replaces per-machine shell deploy scripts with one declarative,
cross-platform (Linux/macOS/Windows) command. Everything it does comes from a
`deploy.yaml` file — the tool itself has no user- or machine-specific assumptions built in.

```bash
agentpm deploy                          # uses ./deploy.yaml in cwd
agentpm deploy --config ./ops/deploy.yaml
agentpm deploy --dry-run                # report planned actions, touch nothing
agentpm deploy --json                   # machine-readable output
```

## Config file

All relative paths inside `deploy.yaml` resolve relative to the **YAML file's own
location**, not the current working directory. A leading `~/` expands to
`os.homedir()`. Absolute paths pass through unchanged.

```yaml
base:
  - src: ./files/claude-settings.json
    dest: ~/.claude/settings.json
    mode: always        # or "if-missing"
    # backup defaults to true for mode: always

consistency:
  library-vs: ~/dev/skills-checkout

instructions:
  - dest: ~/.claude/CLAUDE.md
    header: |
      <!-- Generated by agentpm deploy - do not edit.
           Change the source files instead. -->
    sources:
      - ./instructions/base-rules.md
      - ./instructions/team-conventions.md
    # header/footer are LITERAL inline strings (use YAML | blocks),
    # not file paths. File content belongs in sources - a footer file
    # is simply the last sources entry.

pull:
  from: my-skills
  target: [claude, codex, kimi]
  transform: [codex-agents, kimi-agents]
  agents: true

export:
  - layout: antigravity
    dest: ~/.antigravity/plugins/agentpm
    optional: true
```

Unknown keys anywhere in the file are rejected with an error naming the offending key and,
for list entries, the entry's index (e.g. `Unknown key "wat" in base[0]. Allowed keys: ...`).

### `base`

Each entry copies `src` to `dest`. `src` may be a single file or a directory; when it's a
directory, AgentPM syncs the files **directly inside it** (non-recursive — nested
subdirectories are not synced) into `dest`, matching files by name.

- `mode: always` — copy whenever the source and destination content differ. Before
  overwriting an existing destination file, AgentPM backs it up (unless `backup: false`).
  Unchanged files are left untouched (no copy, no backup).
- `mode: if-missing` — only write when `dest` does not already exist; an existing
  destination is always left as-is.

Backups land in `~/.agentpm/backups/deploy-<ISO-timestamp>/` (colons in the timestamp are
replaced with `-` for Windows compatibility), mirroring the destination's absolute path so
backups from the same run never collide.

### `consistency`

`library-vs: <checkout-dir>` compares every skill's `SKILL.md` in the local AgentPM skill
library (`~/.agentpm/skills/*`) against `<checkout-dir>/skills/<name>/SKILL.md`. Any
difference **aborts the deploy** with a message listing every mismatched or missing skill —
this catches "library edited locally but not pushed" and "checkout is stale" before they
silently diverge across machines.

### `instructions`

Concatenates the literal `header` string (optional), the content of each file in `sources`
(in order), and the literal `footer` string (optional) into `dest`, separated by a single
blank line. `header` and `footer` are inline strings (use YAML `|` blocks for multiline),
NOT file paths - file content belongs in `sources`. The tool never injects timestamps or other
volatile content, so the same inputs always produce byte-identical output. Like `base` with
`mode: always`, an existing `dest` is backed up before being overwritten, and an unchanged
result is left untouched.

### `pull`

Delegates to the same logic as `agentpm pull`, with `yes: true` semantics (no prompts):

- `from` — target id or locator to pull from (same as `agentpm pull --from`).
- `target` — subset of `claude`, `codex`, `kimi`, `generic` to fan skills out into (same
  as `agentpm pull --target`).
- `transform` — a single value or list of `codex-agents` and `kimi-agents`.
- `agents` — also materialize `.claude/agents` entries (default `true`).

### `export`

Delegates to the same logic as `agentpm export --dest`, once per list entry. `optional: true`
skips the entry (recorded in `skipped`, not `actions`) when `dest` does not already exist —
use this for runtime-specific plugin directories that only exist on machines where that
runtime is installed, so `deploy` does not create clutter on machines without it.

## Execution order

`base` → `consistency` → `instructions` → `pull` → `export`. A `consistency` mismatch
aborts before `instructions`/`pull`/`export` run (any `base` step before it has already
applied).

## `--dry-run`

Reports every action that would happen — including per-file `base`/`instructions`
decisions — without writing, backing up, or deleting anything, and without invoking
`pull`/`export` (those are only listed as planned).

## JSON output

```json
{
  "ok": true,
  "action": "deploy",
  "result": {
    "success": true,
    "configPath": "/abs/path/deploy.yaml",
    "dryRun": false,
    "actions": ["base copy ...", "instructions write ...", "pull: 2 skill(s), 1 agent(s) from ...", "export: antigravity -> ... (2 skill(s), 1 agent(s))"],
    "warnings": [],
    "skipped": ["export skip: antigravity -> ... (optional, dest missing)"]
  }
}
```
